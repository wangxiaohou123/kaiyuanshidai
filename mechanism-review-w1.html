P0 · 紧急问题（本周必修）
P0
🔴 pending-push.md 积压未清：3月13日日报至今未发送
pending-push.md 中有3条3月13日日报（多次重复生成）至今未推送给 wangjingjing44。 根本原因：Agent 自动生成了内容，但大象登录态失效，发送卡在登录步骤，且没有失败重试机制，导致内容无限积压。

风险：长期不清理 → 每次对话开始都会触发发送逻辑 → 如某次大象恢复登录，会批量推送3条重复日报，造成刷屏。
🔧 修复方案：今日手动清空 pending-push.md 中过期内容（超过24小时的全部归档），同时给 Agent 增加「超过24小时未发送则自动放弃+归档」逻辑。
P0
🔴 auto-sync.sh 未在今日汇报后执行：day4 尚未推送到 GitHub
今日生成的 day4-briefing-0314.html、weekly-review-w1.html、index.html（日志更新）尚未推送到 GitHub Pages。 外网官网目前展示的仍是昨日版本（Day3），与实际进展不同步。

根本原因：commands.md 中写了「每次生成汇报后必须执行 auto-sync.sh」，但没有在 HTML 生成流程末尾嵌入强制触发步骤，依赖「小美记得执行」，属于人工依赖型规则，不可靠。
🔧 修复方案：①本次立即执行 auto-sync.sh 推送今日产出；②在 commands.md 的交付前检核清单新增一行「✅ auto-sync.sh 已执行」，作为硬性 gate。
P1 · 重要优化（Week 2 完成）
P1
🟡 task-log.md 中「行业日报生成」条目重复6次，噪音过大
3月13日 task-log.md 中「行业日报生成」重复记录了6次（01:19、02:00、02:02、06:33、08:00、12:00），每次内容几乎相同，说明 LaunchAgent 定时任务在重复执行但没有判断「今日是否已经生成过」。

这导致日志可读性极差，CEO无法快速定位有效信息。
🔧 修复方案：Agent 每日执行前检查 task-log 当日是否已有「行业日报生成」记录，有则跳过写入，改为追加「n次触发，已有记录，跳过」一行。
P1
🟡 知识库归档路径混乱：decisions 存在两个位置
决策文件同时存在于：
• ~/.xiaomei-workspace/decisions/（旧路径，有5个文件）
• ~/.xiaomei-workspace/knowledge/decisions/（commands.md 规定的新路径，只有1个README）

导致：查找决策记录需要去两个地方，Agent 自动归档也无法确认写哪里。
🔧 修复方案：将 decisions/ 旧路径文件全部迁移到 knowledge/decisions/，并在 commands.md 删除旧路径引用。统一为 knowledge/ 目录树。
P1
🟡 daily-logs 只有3月11日一个文件，日报归档机制实际未运行
knowledge/daily-logs/ 只有 2026-03-11.md，3月12日和3月13日的日报并未按规范写入该目录（实际上写在了 ~/.xiaomei-workspace/daily-logs/ 根目录）。

存在两套归档路径：知识库内和知识库外，且 Agent 生成的路径与 commands.md 规定的不一致。
🔧 修复方案：统一日报归档到 knowledge/daily-logs/，修改 kaiyuan-agent.py 中的输出路径，补充3月12-14日日报到正确目录。
P2 · 长期改进（月度推进）
P2
🔵 大象登录态无保活机制，发送稳定性依赖手动介入
目前大象消息发送需要手动扫码登录，Session 有效期约数小时到1天。Agent 自动生成的日报在 Session 失效后无法自动恢复，只能写入 pending-push 等待下次手动触发。

这是「24小时工作」目标与「手动登录」现实之间的结构性矛盾。
🔧 长期方案：研究大象 API 或定时自动扫码 + 保活 Cookie 方案，Week 3-4 评估可行性。短期：接受手动确认，但完善失败提醒机制。
P2
🔵 Agent 角色分工与小美主对话职责边界不清晰
目前存在两套班子作业方式：①小美直接在对话中完成（本次汇报）；②kaiyuan-agent.py 后台运行产出。两者产出格式、归档路径、质量标准不一致，CEO收到的内容质量参差不齐。

例如：Agent 后台生成的日报「吏部今日AI服务异常」「AI调用失败」等错误信息会直接出现在日报里，影响专业性。
🔧 长期方案：明确分工——Agent 负责「定时搜集+缓存」，小美负责「加工+质检+交付」，两者不重复，各司其职。Month 2 重构 Agent。
P2
🔵 研究院课题完成率追踪缺乏自动化，每次靠人工回忆
目前 T01-T10 课题状态（完成/进行中/未启动）存在官网 index.html 里，但没有机器可读的进度文件，每次汇报「研究院完成60%」需要人工数文件。

随着课题增多，人工统计误差会加大。
🔧 长期方案：创建 knowledge/research-index.json，每次新增/完成课题时自动更新，weekly-review 自动读取计算百分比。

Mechanism Verdict · 2026-03-14
AI工作系统第一周：内容能力 A，自动化可靠性 C+
🏆
最强的地方：苏格拉底质询 × 三省六部产出 × 珍珠白可视化的组合，形成了高质量的「思考-产出-交付」流水线。这是本系统的核心价值，继续强化。
⚠️
最脆弱的地方：所有自动化步骤（发消息、推GitHub、归档日志）都依赖「小美记得」或「登录态未过期」，缺乏容错设计。一旦某步失败，没有报警，悄悄堆积成积压。
🎯
Week 2 核心改进目标：把 auto-sync.sh 嵌入交付检核清单成为硬性 gate，彻底消灭「汇报生成但未上线」的情况；同时清理 pending-push 积压，建立「失败即归档」的兜底机制。
📐
Month 2 架构方向：「Agent 采集 + 小美加工」双层分工；知识库统一 knowledge/ 目录树；大象保活机制评估。届时整体机制健康度目标：A级。
